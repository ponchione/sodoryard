### What I found

Primary files:
- `Tool.ts`
- `services/tools/toolExecution.ts`
- `utils/toolResultStorage.ts`
- `constants/toolLimits.ts`
- `utils/task/outputFormatting.ts`

- `services/compact/microCompact.ts`
- `services/toolUseSummary/toolUseSummaryGenerator.ts`
- `tools/FileReadTool/FileReadTool.ts`
- `tools/FileReadTool/limits.ts`
- `utils/messages.ts`

Key functions/constants:
- `Tool.maxResultSizeChars` in `Tool.ts`
- `renderToolResultMessage()`, `extractSearchText()`, `isResultTruncated()`, `getToolUseSummary()` in `Tool.ts`
- `addToolResult()` in `services/tools/toolExecution.ts`
- `processToolResultBlock()`, `processPreMappedToolResultBlock()`, `maybePersistLargeToolResult()`, `persistToolResult()`, `buildLargeToolResultMessage()`, `generatePreview()`, `enforceToolResultBudget()`, `applyToolResultBudget()` in `utils/toolResultStorage.ts`
- `DEFAULT_MAX_RESULT_SIZE_CHARS = 50_000` and `MAX_TOOL_RESULTS_PER_MESSAGE_CHARS = 200_000` in `constants/toolLimits.ts`
- `TASK_MAX_OUTPUT_DEFAULT = 32_000`, `TASK_MAX_OUTPUT_UPPER_LIMIT = 160_000` in `utils/task/outputFormatting.ts`
- `TIME_BASED_MC_CLEARED_MESSAGE = "[Old tool result content cleared]"` and `COMPACTABLE_TOOLS` in `services/compact/microCompact.ts`

### How it works

CC does not just naively clip tool output and stuff it into history.

There are two distinct layers:

1. Per-tool result handling
- Each tool can declare `maxResultSizeChars`.
- Tool results flow through `processPreMappedToolResultBlock()` / `processToolResultBlock()` before being emitted as `tool_result` blocks.
- If a result is too large, `maybePersistLargeToolResult()` writes the full output to session storage and replaces the model-visible content with a wrapper containing:
  - a `<persisted-output>` tag
  - the stored file path
  - a preview generated by `generatePreview()`
- Preview size is bounded by `PREVIEW_SIZE_BYTES = 2000`.
- Empty output is normalized to `"(toolName completed with no output)"` to avoid bad tail/stop-sequence behavior.

2. Aggregate per-message budget enforcement
- Even when individual tool outputs are under their own thresholds, a merged user message containing many parallel tool results can still become too large.
- `enforceToolResultBudget()` groups candidates the same way messages will later be normalized for the API, then persists the largest fresh tool results until the combined char budget is under `MAX_TOOL_RESULTS_PER_MESSAGE_CHARS`.
- Replacement decisions are memoized by `tool_use_id` so the same result is replaced the same way across replay/resume/fork paths.

Shell/task output gets its own formatting path:
- `TaskOutput` spills large stdout/stderr to disk.
- `formatTaskOutput()` prepends a truncation notice and keeps the tail that fits, explicitly favoring the end of output where errors usually live.

There is also later-stage cleanup:
- `microCompact.ts` can clear older compactable tool results after inactivity, replacing them with `[Old tool result content cleared]`.
- `query.ts` launches async tool-use summarization for previous turns.
- `services/toolUseSummary/toolUseSummaryGenerator.ts` uses a Haiku model to compress batches of tool usage into short labels, with JSON input/output truncated before prompting.

### The interesting bits

- The strongest pattern is not “truncate harder”; it is “persist full output externally, show a preview and stable reference to the model.”
- CC separately handles per-result budget and aggregate message budget. That is more robust than a single global truncation rule.
- `FileReadTool` opts out of persisted-output replacement by setting `maxResultSizeChars: Infinity`; CC treats “persist a read result to a file, then read that file” as a circular anti-pattern.
- Result replacement decisions are stabilized specifically to protect prompt cache prefix identity, not just for replay correctness.
- The shell/task formatter keeps the tail, not just the head, which matches where failures cluster in real logs.
- CC does use model-based summarization, but mostly for tool-use summaries / compaction layers, not as the first-line response to every large tool output.

### Relevance to sirtopham

This strongly validates sirtopham’s two-phase idea, but suggests a more specific architecture:
- Keep per-tool truncation.
- Add a separate aggregate “all tool results in this merged message” budget.
- Prefer persisted full outputs + preview/reference over pure truncation for many large tool classes.
- Preserve tail-heavy formatting for shell/test/build outputs.
- Treat file-read results specially rather than applying the same persistence policy blindly.

## 2. System Prompt Engineering

### What I found

Primary files:
- `constants/prompts.ts`
- `constants/systemPromptSections.ts`
- `context.ts`
- `utils/queryContext.ts`
- `QueryEngine.ts`
- `tools.ts`
- `Tool.ts`

Key functions/constants:
- `getSystemPrompt()` in `constants/prompts.ts`
- `SYSTEM_PROMPT_DYNAMIC_BOUNDARY = "__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__"`
- `systemPromptSection()` and `DANGEROUS_uncachedSystemPromptSection()` in `constants/systemPromptSections.ts`
- `fetchSystemPromptParts()` in `utils/queryContext.ts`
- `getUserContext()` and `getSystemContext()` in `context.ts`

### How it works

The prompt is assembled as a stable cached prefix plus a dynamic tail.

`getSystemPrompt()` builds the base prompt from named sections, including:
- intro/personality
- system behavior
- task execution guidance
- action guidance
- tool-usage guidance
- tone/style
- output efficiency

Dynamic prompt sections are managed through section registries and include things like:
- `session_guidance`
- memory prompt content
- model override hints
- environment info
- language info
- output style
- `mcp_instructions` (explicitly uncached)
- scratchpad-related instructions
- function-result clearing
- summarize-tool-results behavior
- token budget info
- brief mode instructions

`fetchSystemPromptParts()` in `utils/queryContext.ts` assembles:
- default system prompt or `customSystemPrompt`
- optional user context
- optional system context
- optional appended system prompt

If a custom system prompt is provided, CC intentionally skips the normal default prompt and system context.

The prompt text itself encodes a lot of behavioral policy:
- prefer dedicated tools over Bash
- parallelize tool calls when possible
- do not blindly retry denied tool calls
- treat tool output as untrusted / prompt-injection-capable
- require confirmation for risky actions

### The interesting bits

- CC is very careful about prompt byte stability. `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` exists partly to support cache-sensitive assembly rules.
- Dynamic sections are explicitly categorized as cacheable vs intentionally uncached.
- `Tool.ts` stores rendered system prompt bytes in tool-use context so forked subagents can inherit identical prompt bytes and preserve cache sharing.
- Tool descriptions matter operationally: a lot of “behavior shaping” is in tool metadata and section text, not only the top-level role framing.
- There are code paths to avoid leaking model names/IDs in certain “Undercover” builds.

### Relevance to sirtopham

This strongly supports sirtopham’s Block 1 / Block 2 separation. The biggest actionable lessons are:
- keep a hard boundary between stable and volatile prompt sections
- classify prompt sections by cacheability deliberately
- treat tool schema text as part of prompt engineering, not just API plumbing
- stabilize prompt bytes for subagents/forks if cache reuse matters

## 3. Agent Loop Termination and Convergence

### What I found

Primary files:
- `query.ts`
- `QueryEngine.ts`
- `query/stopHooks.ts`
- `query/tokenBudget.ts`
- `constants/prompts.ts`
- `utils/permissions/denialTracking.ts`

Key functions/constants:
- max-turn handling in `query.ts`
- `createBudgetTracker()` / `checkTokenBudget()` in `query/tokenBudget.ts`
- denial thresholds in `utils/permissions/denialTracking.ts`
- structured-output retry handling in `QueryEngine.ts`

### How it works

The main hard stop is a configurable turn/iteration cap in `query.ts`.
- When another round would exceed `maxTurns`, CC emits a `max_turns_reached` attachment and terminates with reason `max_turns`.

But convergence is managed by several softer mechanisms around that core loop:
- token/task budget tracking via `createBudgetTracker()` and `checkTokenBudget()`
- structured-output retry cap in `QueryEngine.ts` (default environment-backed limit of 5 retries)
- stop hooks in `query/stopHooks.ts` that can prevent continuation
- permission/classifier denial counters with thresholds:
  - `maxConsecutive: 3`
  - `maxTotal: 20`

There is also convergence-by-history-shaping:
- after compaction boundaries, `QueryEngine.ts` splices old message arrays to drop pre-compaction history from active memory and keep long sessions manageable.

### The interesting bits

- I did not find a rich semantic loop detector of the kind “same plan, different wording” or similarity-based thrash detection.
- The soft signals are distributed across neighboring systems: token budgets, permission-denial limits, structured-output retry caps, and stop hooks.
- CC appears to rely more on prompting + compaction + specialized guardrails than on an explicit “you are looping” injected meta-prompt.
- I did not see a “disable tools on final iteration” mechanism matching sirtopham’s design.

### Relevance to sirtopham

sirtopham’s same-tool+args failure detector is still useful; CC does not seem to have an obviously superior semantic loop detector here. The more interesting lesson is that convergence can be improved by adding orthogonal soft stops:
- structured-output retry caps
- denial-count caps
- token/task budget exhaustion signals
- stop hooks that can halt continuation cleanly

That suggests sirtopham may want middle-ground heuristics beyond repeated-tool failure, but not necessarily a heavyweight semantic loop detector first.

## 4. File Edit Mechanics

### What I found

Primary files:
- `tools/FileEditTool/FileEditTool.ts`
- `tools/FileEditTool/utils.ts`
- `tools/FileEditTool/types.ts`
- `tools/FileEditTool/constants.ts`
- `tools/FileWriteTool/FileWriteTool.ts`
- `tools/NotebookEditTool/NotebookEditTool.ts`

Key constants/functions:
- `FILE_EDIT_TOOL_NAME = 'Edit'`
- `FILE_UNEXPECTEDLY_MODIFIED_ERROR`
- `validateInput()` and `call()` in `FileEditTool.ts`
- `findActualString()`, `normalizeQuotes()`, `preserveQuoteStyle()`, `applyEditToFile()`, `getPatchForEdits()`, `normalizeFileEditInput()` in `tools/FileEditTool/utils.ts`
- `DIFF_SNIPPET_MAX_BYTES = 8192`

### How it works

CC’s edit tool is exact-match-based, but not purely brittle byte matching.

Input schema includes:
- `file_path`
- `old_string`
- `new_string`
- `replace_all`

Validation path:
- normalize path with `expandPath()`
- deny-rule permission check
- reject `old_string === new_string`
- reject UNC paths on Windows for safety
- read file directly, detect encoding/BOM, normalize CRLF to LF
- if file does not exist, creation through Edit is only allowed when `old_string === ''`
- reject `.ipynb` and redirect to `NotebookEdit`
- require a prior full read via `readFileState`; partial reads are not sufficient
- perform stale-write detection using cached read timestamp and file mtime
- on Windows, allow through false-positive mtime churn if actual content matches cached content
- resolve `old_string` via `findActualString()` which can normalize quote variants
- reject zero matches; reject multiple matches unless `replace_all=true`

Application path:
- create parent dirs if needed
- perform file-history tracking
- re-check stale-write conditions inside the critical section
- preserve quote style in replacements via `preserveQuoteStyle()`
- generate patch output with `getPatchForEdit()` / `getPatchForEdits()`
- write back preserving encoding and line endings
- clear/update diagnostics and notify editor integrations
- refresh `readFileState`
- optionally include git diff data in output

`FileWriteTool` exists alongside `Edit` and performs whole-file replacement with similar read-before-write and stale-write protections.

### The interesting bits

- The contract is “exact match, but with quote normalization and some de-sanitization help,” not “regex” or “fuzzy diff patching.”
- Edit refuses to operate after partial reads. That is a strong safety invariant.
- CC repeats stale-write checks just before writing, not only during preflight validation.
- Edit can create new files, but only under a very narrow empty-`old_string` contract.
- There is a dedicated notebook editing path rather than forcing notebooks through text edit semantics.
- Quote preservation is surprisingly deliberate; this is battle-tested polish rather than a minimal string replace.

### Relevance to sirtopham

This mostly validates sirtopham’s exact-one-match design, but suggests a few upgrades:
- require full-read-before-edit, not just arbitrary file visibility
- keep a second stale-write check immediately before disk write
- add limited normalization helpers for common LLM drift cases like curly/straight quotes
- keep a separate whole-file write tool
- consider a notebook-specific tool if notebooks matter

## 5. Cancellation and Partial State Cleanup

### What I found

Primary files:
- `utils/abortController.ts`
- `hooks/useCancelRequest.ts`
- `services/tools/StreamingToolExecutor.ts`
- `services/tools/toolExecution.ts`
- `hooks/toolPermission/PermissionContext.ts`
- `query.ts`
- `query/stopHooks.ts`
- `utils/messages.ts`
- `utils/handlePromptSubmit.ts`
- `cli/print.ts`

Key functions/constants:
- `createAbortController()`, `createChildAbortController()` in `utils/abortController.ts`
- `handleCancel()` in `hooks/useCancelRequest.ts`
- `cancelAndAbort()` in `PermissionContext.ts`
- `createToolResultStopMessage()`, `createUserInterruptionMessage()`, `CANCEL_MESSAGE` in `utils/messages.ts`
- `discard()`, `getAbortReason()`, `executeTool()` in `StreamingToolExecutor.ts`

### How it works

Cancellation is built around abort-controller hierarchies.
- `createChildAbortController(parent)` propagates parent aborts to children and uses cleanup/WeakRef patterns to avoid retention leaks.

User cancellation entrypoints include:
- explicit cancel requests
- interrupting with a new prompt submission
- CLI interrupt pathways

Permission flow can abort execution too:
- a denial in `PermissionContext.ts` can both resolve a permission decision and abort the current tool/turn.

Tool execution path:
- if a tool is already aborted before execution, CC synthesizes an error `tool_result` instead of silently dropping it
- `StreamingToolExecutor` gives each tool a child abort controller under a sibling controller under the turn controller
- if a Bash tool errors, sibling Bash/tool execution may be aborted with reason `sibling_error`
- user `interrupt` is special-cased: only tools whose `interruptBehavior() === 'cancel'` get cancelled on prompt-interrupt

Query-level cleanup:
- if abort occurs during model streaming, CC drains remaining streaming tool results or synthesizes missing `tool_result` blocks so tool-use / tool-result pairing stays valid
- failed streaming attempts are tombstoned; old partial assistant state arrays are cleared before retry/fallback
- stop hooks also poll abort state and can emit interruption messages before halting continuation

### The interesting bits

- The distinction between generic abort and `interrupt` is important. `interrupt` means “the user moved on with another prompt,” not just “stop everything noisily.”
- CC does real orphan cleanup: it tombstones partial assistant messages and synthesizes missing tool results to keep transcript structure coherent.
- Permission rejection can bubble from per-tool child abort back to the turn-level controller.
- Bash failures are treated specially enough to cancel siblings; other tool errors usually are not.
- I did not find a rollback mechanism for filesystem writes. Cleanup is focused on transcript/tool-state coherence, not undoing completed file mutations.

### Relevance to sirtopham

This broadly validates sirtopham’s context-based cancellation and completed-iteration persistence rules. The key additions worth stealing are:
- hierarchical abort controllers
- synthesized missing tool results to preserve message-pair invariants
- explicit tombstoning/reset of partial assistant state after stream failure
- distinguishing “interrupt due to new prompt” from generic cancellation

Filesystem rollback still appears to be an open problem in CC as well.

## 6. Token Counting and Context Window Management

### What I found

Primary files:
- `utils/tokens.ts`
- `utils/context.ts`
- `query.ts`
- `query/tokenBudget.ts`
- `services/compact/autoCompact.ts`
- `services/compact/compact.ts`
- `tools/FileReadTool/limits.ts`
- `tools/FileReadTool/FileReadTool.ts`
- `services/api/claude.ts`

Key functions/constants:
- `tokenCountWithEstimation()` in `utils/tokens.ts`
- `finalContextTokensFromLastResponse()` in `utils/tokens.ts`
- `MODEL_CONTEXT_WINDOW_DEFAULT = 200_000` in `utils/context.ts`
- `getContextWindowForModel()` and `getModelMaxOutputTokens()` in `utils/context.ts`
- `getAutoCompactThreshold()` and `shouldAutoCompact()` in `services/compact/autoCompact.ts`
- `createBudgetTracker()` and `checkTokenBudget()` in `query/tokenBudget.ts`

### How it works

CC has a real token accounting layer rather than a simple chars/4 heuristic everywhere.

Canonical count path:
- `tokenCountWithEstimation()` is the main threshold-checking function.
- It uses actual usage from the latest assistant/API response when available, then estimates only the subsequent messages that were not part of that response.
- It handles split assistant records by walking back to the first sibling with the same assistant message ID.

Context window management:
- `getContextWindowForModel()` derives effective window from model capabilities, optional `[1m]` suffixes, experiment gates, and environment overrides.
- `getMaxOutputTokensForModel()` / `getModelMaxOutputTokens()` cap output-token budgets with additional slot-reservation logic.

Compaction thresholds:
- `autoCompact.ts` calculates effective context window after reserving summary output space.
- Auto-compaction and warning/error thresholds are based on token counts, not just chars.
- `query.ts` orchestrates multiple steps before/around API calls:
  - tool result budgeting
  - snip compaction
  - microcompact
  - context collapse
  - auto-compact
  - reactive compact on overflow/media errors

File reads have their own token gate:
- `FileReadTool` uses rough file-type-based estimation first
- if near threshold, it calls `countTokensWithAPI(content)` for a better check
- `maxResultSizeChars` for file read is `Infinity`; token limits are the true bound there

### The interesting bits

- CC’s “canonical token count” is hybrid: reuse real API usage where possible, estimate only the delta after that. That is much better than global rough estimation.
- They reserve explicit space for summary/output generation when computing effective context capacity.
- Compaction has several layers, not one compression trigger.
- File-read counting uses a cheap-first, expensive-only-near-risk strategy.
- I did not find evidence of a bundled local tokenizer being the primary universal source of truth; API usage and mixed estimation appear central.

### Relevance to sirtopham

This challenges sirtopham’s current chars/4 heuristic in a useful way. A likely upgrade path is:
- keep chars/4 as cheap preflight if needed
- but promote a hybrid count function that reuses actual API usage from the last round and only estimates subsequent growth
- reserve explicit response/summary budget when deciding compression thresholds
- treat file reads with a separate token-aware gate

## 7. Streaming Event Architecture

### What I found

Primary files:
- `services/api/claude.ts`
- `query.ts`
- `QueryEngine.ts`
- `entrypoints/sdk/coreSchemas.ts`
- `utils/messages.ts`
- `hooks/useRemoteSession.ts`
- `cli/structuredIO.ts`

Key concepts:
- raw stream events: `message_start`, `content_block_start`, `content_block_delta`, `content_block_stop`, `message_delta`, `message_stop`
- outward normalized event type: `stream_event`

### How it works

There are multiple layers of event handling.

1. Anthropic stream parsing in `services/api/claude.ts`
- Raw SSE-style model stream chunks are parsed into message/content-block structures.
- `input_json_delta` grows partial tool input JSON.
- `text_delta`, thinking deltas, and signatures are appended to mutable partial structures.

2. Query loop in `query.ts`
- Emits `stream_request_start` before model streaming.
- Consumes partial assistant messages as they materialize.
- Extracts `tool_use` blocks and hands them to `StreamingToolExecutor`.
- Interleaves completed tool results/progress with continued model streaming when possible.

3. QueryEngine normalization
- `QueryEngine.ts` tracks usage and stop reasons across stream deltas.
- It yields outward `stream_event` records when partial streaming is enabled.
- SDK schema for these events is in `entrypoints/sdk/coreSchemas.ts`.

4. UI/remote live-state reconstruction
- `utils/messages.ts` maps stream events into UI modes like:
  - requesting
  - thinking
  - responding
  - tool-input
  - tool-use
- `hooks/useRemoteSession.ts` feeds remote stream events through the same live reconstruction logic.

### The interesting bits

- `stream_event` is basically a live side-channel; durable transcript state is still stored as normal assistant/user/progress/system/attachment messages.
- Tool input streaming is supported via `input_json_delta`, not just text token streaming.
- Completed tool results can interleave with model output in the same turn rather than waiting for the full model response to finish.
- `cli/structuredIO.ts` uses a single outbound queue to preserve ordering between stream events and control/permission messages.
- VCR/test recording skips `stream_event`, which underscores that partials are for UX, not durable state.

### Relevance to sirtopham

This is directly relevant to your “incremental tool output or one block?” question. CC’s architecture supports interleaving and live progress, but still keeps the durable transcript model simple. For sirtopham, a good pattern may be:
- keep durable turn state coarse and typed
- stream partial/live events separately
- support streaming tool input/progress if UX benefits justify it
- ensure strict ordering on the output channel

## 8. Error Recovery Patterns

### What I found

Primary files:
- `services/api/withRetry.ts`
- `services/api/claude.ts`
- `services/api/errors.ts`
- `QueryEngine.ts`
- `query.ts`
- `query/stopHooks.ts`
- `entrypoints/init.ts`
- `remote/SessionsWebSocket.ts`

Key behaviors:
- exponential backoff with jitter in `withRetry.ts`
- `Retry-After` support
- context overflow parsing via `parseMaxTokensContextOverflowError()`
- stream-idle timeout fallback in `services/api/claude.ts`

### How it works

API-level recovery:
- `withRetry.ts` classifies retryable cases and applies exponential backoff with jitter.
- It honors `Retry-After` headers and has a persistent retry mode for 429/529 classes.
- In persistent retry mode, long waits are chunked and heartbeat-style `api_error` system messages are emitted so hosts do not think the session died.
- Context overflow errors are parsed and treated as recoverable.

Streaming recovery:
- If the stream goes idle too long, CC throws and falls back to a non-streaming retry path.
- If a stream ends without proper `message_start` / completed content / stop reason, that also triggers fallback.
- `model_context_window_exceeded` is normalized into max-output-token recovery behavior.
- refusal stop reasons are mapped to user-facing refusal guidance.

Other recovery layers:
- orphaned permission requests are deduplicated and recovered in both `QueryEngine.ts` and `cli/print.ts`
- remote websocket session logic retries certain stale-session failures because compaction can temporarily invalidate assumptions
- stop hooks swallow non-critical cleanup errors so they do not destabilize the main path

### The interesting bits

- CC distinguishes transport/stream weirdness from model refusal from context overflow, and recovers differently for each.
- Heartbeat messages during persistent retry are a nice product detail: they keep hosts/UI from misclassifying the session as frozen.
- Some retries are designed specifically to preserve prompt cache identity by keeping key request parameters stable.
- Hook failures are often downgraded instead of being allowed to poison the main task flow.

### Relevance to sirtopham

This validates sirtopham’s three-tier framing. The strongest additions would be:
- explicit stream-idle detection with non-streaming fallback
- heartbeat/progress events during long API retries
- special handling for malformed/incomplete stream termination
- careful parameter stability on retries to preserve cache hits

## 9. Permission and Safety Model

### What I found

Primary files:
- `types/permissions.ts`
- `utils/permissions/permissionSetup.ts`
- `utils/permissions/permissions.ts`
- `utils/permissions/yoloClassifier.ts`
- `utils/permissions/filesystem.ts`
- `utils/permissions/denialTracking.ts`
- `utils/permissions/classifierDecision.ts`
- `setup.ts`

Key concepts:
- modes: `acceptEdits`, `bypassPermissions`, `default`, `dontAsk`, `plan`, plus internal `auto` and `bubble`
- auto-mode dangerous rule stripping
- classifier-based approval for ambiguous operations

### How it works

Startup and mode gates:
- `setup.ts` blocks especially dangerous bypass configurations when running as root/sudo outside safer environments.
- `transitionPermissionMode()` centralizes mode transitions.
- Entering auto mode strips dangerous pre-authorized rules; leaving restores them.

Permission decision path:
- `hasPermissionsToUseTool()` in `utils/permissions/permissions.ts` is the core decision pipeline.
- `dontAsk` can convert asks to deny.
- auto/plan+auto can route asks into the classifier path.
- hooks can resolve permission requests before UI prompts.

Classifier model:
- `yoloClassifier.ts` builds a dedicated system prompt with environment, allow/deny rules, and even CLAUDE.md content wrapped as user intent.
- It supports a two-stage XML classifier: fast pass, then a thinking pass for blocked cases.
- If the classifier transcript grows too long:
  - interactive mode falls back to manual approval
  - headless mode aborts

Safety details:
- `filesystem.ts` classifies paths as safe / unsafe / classifier-approvable / manual-only
- some path classes are checked before user allow rules to prevent accidental pre-approval of especially sensitive files
- `classifierDecision.ts` includes a safe allowlist of read/search/task/UI-ish operations that bypass the classifier

### The interesting bits

- Auto mode is not a cosmetic toggle; it rewrites active permission state by stripping dangerous allow rules that would bypass the classifier.
- CC distinguishes “classifier-approvable” from “manual-only” operations.
- The classifier uses user context files as user-role input, not system-role input.
- Failure posture is configurable: some environments fail closed, some can fail open.
- There is much more machinery here than sirtopham likely wants, but it reveals where Anthropic found real sharp edges.

### Relevance to sirtopham

Even if sirtopham remains deliberately lighter-weight, a few CC patterns look worth copying:
- keep project/path safety checks separate from general tool permission checks
- distinguish safe reads from risky mutations clearly
- consider special handling for sensitive config locations even if there is no user approval flow
- think about what “fail closed” should mean if any automated safety check becomes unavailable

## 10. Prompt Caching Implementation

### What I found

Primary files:
- `services/api/claude.ts`
- `services/compact/microCompact.ts`
- `bootstrap/state.ts`
- `constants/prompts.ts`
- `constants/systemPromptSections.ts`
- `Tool.ts`
- `services/AgentSummary/agentSummary.ts`
- `utils/sideQuestion.ts`

Key functions/constants:
- `getCacheControl()` and `should1hCacheTTL()` in `services/api/claude.ts`
- `addCacheBreakpoints()` in `services/api/claude.ts`
- `consumePendingCacheEdits()`, `pinCacheEdits()`, `markToolsSentToAPIState()`, `resetMicrocompactState()` in `services/compact/microCompact.ts`
- cache latching fields in `bootstrap/state.ts`

### How it works

CC uses Anthropic prompt caching very deliberately.

Basic cache control:
- `getCacheControl()` returns ephemeral cache-control objects, optionally with TTL/scope.
- session-level latches in `bootstrap/state.ts` keep cache-relevant headers/settings sticky to avoid mid-session cache busts.

Breakpoint placement:
- `addCacheBreakpoints()` inserts exactly one message-level `cache_control` marker per request.
- If `skipCacheWrite=true`, it uses the second-to-last message so fire-and-forget/fork calls can share prefix cache without writing fork tails back into cache.

Cached microcompact / cache editing:
- `microCompact.ts` can prepare `cache_edits` blocks that delete old tool-result content from cached prefixes without mutating the local in-memory transcript.
- `services/api/claude.ts` consumes pending cache edits, inserts `cache_edits`, and adds `cache_reference` markers to earlier `tool_result` blocks.
- Pinned edits are reinserted at stable positions to preserve cache-key identity.

Cross-flow cache reuse:
- fork/subagent paths reuse parent rendered system prompt bytes
- side-question and summary-generation paths are careful not to perturb cache-key-relevant request parameters
- some calls explicitly keep tool definitions present for cache-key compatibility even when tools are effectively disabled

### The interesting bits

- “Exactly one message-level cache_control marker per request” is a strong, explicit design rule.
- Cache editing is used as a way to shrink what the model sees while preserving cacheable prefix structure, which is more advanced than simple compaction.
- CC is obsessed with byte stability: TTL eligibility, fast-mode flags, AFK-mode flags, cache-editing headers, and prompt assembly are all latched to avoid silent cache misses.
- They even avoid time-relative rendering in some prompt-related paths because byte drift would bust cache.

### Relevance to sirtopham

This largely validates sirtopham’s breakpoint-oriented design. The actionable deltas are:
- treat cache stability as a first-class invariant across subagents/forks/retries
- use a very small number of explicit cache breakpoints
- consider cache-edit-like techniques for removing stale tool output from the prompt without reshaping the whole prefix
- ensure tool-definition stability really is stable, because CC clearly treats tool inventory as cache-key-critical

### Streaming transport: transport details were underspecified

The earlier streaming section described the event normalization layer, but the transport detail from the codebase is more specific:
- `cli/transports/SSETransport.ts` implements a transport that reads via Server-Sent Events and writes via HTTP POST.

- So the correct summary is not “CC uses one transport,” but “CC supports multiple transports depending on runtime path, while normalizing them into the same higher-level structured event stream.”
- SSE transport also tracks `Last-Event-ID` / sequence numbers and has explicit liveness timeouts; WebSocket transport tracks last activity and buffered replay.

This matters for sirtopham because the streaming architecture question is partly separable into two layers:
1. transcript/event semantics
2. network transport semantics

CC is quite modular between those layers.

### File edit mechanics: exact schema wording and self-correction cues

The first draft covered the mechanics, but a few schema/error-contract details are worth recording explicitly:
- `tools/FileEditTool/types.ts` defines a strict object schema with:
  - `file_path`: “The absolute path to the file to modify”
  - `old_string`: “The text to replace”
  - `new_string`: “The text to replace it with (must be different from old_string)”
  - `replace_all`: “Replace all occurrences of old_string (default false)”
- The tool description itself is terse (`"A tool for editing files"`); most of the real behavior shaping lives in validation rules and returned error messages, not a long natural-language tool description.
- Validation errors include concrete self-correction hints such as:
  - nonexistent file → current working directory plus “Did you mean …?” suggestions

  - multiple matches → tells the model either to set `replace_all=true` or provide more context to uniquely identify the instance

That is a notable design choice: CC puts a lot of teaching pressure on deterministic error messages instead of elaborate tool prose.

## Bottom line for sirtopham

The biggest cross-cutting lesson is that CC’s mature design is less about one magical loop heuristic and more about careful edge-management at every seam:
- stabilize bytes for cache reuse
- budget tool results both individually and in aggregate
- persist oversized outputs with previews instead of only truncating
- preserve transcript invariants on cancellation/fallback
- make edit tools conservative with read-before-write and stale-write checks
- use hybrid real-usage-plus-estimation token accounting

The places where CC most clearly challenges sirtopham’s current design are:
1. aggregate tool-result budgeting, not just per-tool truncation
2. hybrid token counting instead of chars/4 as the main threshold mechanism
3. stronger stale-write/full-read invariants for file editing
4. more explicit partial-state cleanup around aborted streaming/tool execution

The places where CC mostly validates sirtopham are:
1. stable prompt blocks and cache-aware assembly
2. two-phase compression/compaction thinking
3. exact-match file editing with strong self-correction feedback
4. context-based cancellation and process cleanup
5. typed streaming event architecture separate from durable transcript state
