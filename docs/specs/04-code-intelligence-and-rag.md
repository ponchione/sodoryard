# 04 — Code Intelligence & RAG Pipeline

**Status:** Draft v0.1 — LanceDB Evaluation Complete **Last Updated:** 2026-03-27 **Author:** Mitchell

---

## Overview

The code intelligence layer is sirtopham's differentiator. It provides semantic understanding of the codebase via tree-sitter parsing, Go AST analysis, and vector-based retrieval, enabling the context assembly layer ([[06-context-assembly]]) to fetch exactly the right code for each conversation turn.

This document covers the parsing pipeline, embedding strategy, vector storage, indexing lifecycle, query/retrieval flow, and the structural graph that complements RAG.

---

## Evaluation Summary (2026-03-27)

**Verdict: The topham RAG pipeline is solid. Carry it forward.**

The pipeline was audited end-to-end from the agent-conductor codebase. The architecture is well-designed, the chunking strategy is correct (semantic boundaries, not arbitrary text windows), and LanceDB is being used appropriately as a straightforward vector store. No technology swap is needed.

Key strengths identified:

- Chunks are cut at function/method/type boundaries via tree-sitter and Go AST analysis
- Embeddings are built from `signature + description` (semantic intent, not raw code)
- Descriptions are generated by a local LLM, capturing what a function does rather than how
- Multi-query expansion with hit-count re-ranking in the searcher
- One-hop call graph expansion (direct hits + their callers/callees)
- Three-pass indexing: parse → build reverse call graph → describe + embed + store

The quality bottleneck, if one exists, is in the local LLM's description quality — not in the vector store or retrieval mechanics.

## Architecture: Two Complementary Systems

sirtopham carries forward two code intelligence systems from topham. They serve different purposes and should both be preserved.

### RAG (Semantic Search)

"Find code that's _about_ this concept."

Given a natural language query, returns the most semantically relevant code chunks. Powered by embeddings and vector similarity search. Good for exploratory queries: "authentication logic", "how does routing work", "error handling patterns."

### Structural Graph (Blast Radius)

"Find code that's _connected to_ this symbol."

Given a specific symbol (function, type), traverses the call graph to find upstream callers, downstream callees, and interface relationships. Powered by Go AST / tree-sitter analysis stored in SQLite. Good for impact analysis: "what would break if I change this function", "what calls this handler."

In sirtopham's context assembly, RAG answers "what's relevant to this question" and the graph answers "what's structurally related to these files."

## Component: Parsing Pipeline

### Tree-sitter Parsers

Used for TypeScript/TSX, Python, and as a fallback for Go (when go/packages analysis fails).

Extracts top-level declarations:

- **Go:** function_declaration, method_declaration, type_declaration
- **TypeScript:** function/class/interface/type_alias/enum declarations, exported arrow functions
- **Python:** function/class definitions
- **Markdown:** splits on `##` headers into sections
- **Fallback:** 40-line sliding windows with 20-line overlap (for unsupported languages)

Each parsed element produces a `RawChunk` with: name, signature (everything before the body), full body text, chunk type, line range.

### Go AST Parser (Enhanced)

For Go files, a richer parser using `go/packages` and `go/types`:

- Full type resolution across the project
- **Call graph extraction:** which functions each function calls (via AST walking of call expressions)
- **Type usage tracking:** which types are referenced in definitions
- **Interface implementation detection:** which interfaces each type satisfies
- **Import tracking:** which packages each file imports

This metadata powers both the reverse call graph construction and the structural graph's blast radius analysis.

The Go AST parser is initialized once per indexing run (loads all packages — takes seconds) and is then cheap per-file.

### Chunk Constraints

- `MaxBodyLength = 2000` characters. Bodies exceeding this are truncated before storage.
- Chunk ID is deterministic: `sha256(filePath + chunkType + name + lineStart)`. Prevents collisions for same-named symbols.
- Content hash (`sha256(body)`) enables change detection for incremental indexing.

## Component: Description Generation

The `Describer` sends file content to a local LLM and receives back a JSON array of `{name, description}` pairs — one per function/type in the file. One LLM call per file.

Key details:

- File content is truncated to 6000 characters before sending
- Relationship context (calls, called_by, types_used, implements) is appended to file content to help the LLM understand the function's role
- Descriptions are 1-2 sentence semantic summaries of what each function does
- If the LLM call fails or returns invalid JSON, the file is still indexed — just without descriptions

**Quality risk:** Description quality depends entirely on the local LLM. Bad descriptions → bad embeddings → bad retrieval. This is the most important variable to monitor. If a particular local model produces vague or inaccurate descriptions, switching to a better model (or even using a cloud model for description generation) would have more impact than any change to the vector store.

## Component: Embedding Pipeline

### Model

nomic-embed-code running in a Docker container (port 8081), accessed via the standard `/v1/embeddings` API endpoint.

### What Gets Embedded

Not raw function bodies. The embedding text for each chunk is:

```
{signature}\n{description}
```

This means the vector space captures the function's interface and semantic purpose. A query like "auth middleware" matches functions described as handling authentication, even if the code never contains the word "auth."

### Query Embedding

Queries are prefixed with: `"Represent this query for searching relevant code: "` — this is nomic-embed-code's recommended query prefix for asymmetric retrieval.

### Dimensions

`DefaultEmbeddingDims = 3584` — high dimensionality, expressive vectors. Storage and search are slightly heavier but quality is better.

### Batching

`DefaultEmbedBatchSize = 32` texts per request. Larger files with many chunks are automatically split into multiple batch requests.

## Component: Vector Store (LanceDB)

### Why LanceDB Is Fine

LanceDB is used as a straightforward vector store with a clean schema. The integration is correct, the Arrow record construction is proper, and there are no LanceDB-specific issues causing quality problems. At codebase scale (thousands to tens of thousands of chunks), any vector store would perform comparably.

### Schema

A single `chunks` table with columns for: identity (id, project_name), location (file_path, language, chunk_type), content (name, signature, body, description), metadata (line_start, line_end, content_hash, indexed_at), relationships (calls, called_by, types_used, implements_ifaces, imports — stored as JSON strings), and the embedding vector (float32, 3584 dimensions).

### Operations

- **Upsert:** delete-by-ID then insert (LanceDB doesn't have native upsert)
- **Vector search:** cosine similarity with optional metadata filters (language, chunk_type, file_path prefix)
- **Metadata queries:** select by file_path or name for change detection and call graph lookups
- **Schema versioning:** `SchemaVersion` constant triggers full re-index on change

### Simplification Option (Not Recommended Now)

The LanceDB API surface is minimal enough that sqlite-vec could be a drop-in replacement, consolidating everything into SQLite. This would eliminate one CGo dependency. However, since CGo is already required for tree-sitter, this is a lateral move with no quality benefit. Keep in back pocket for future simplification if desired.

## Component: Indexing Pipeline

### Three-Pass Architecture

**Pass 1 — Walk + Parse:**

- Walk the project directory filtered by include/exclude globs from config
- Skip files whose content hash hasn't changed since last index
- Parse each file with the appropriate parser (Go AST for Go, tree-sitter for others)
- Produce `Chunk` objects with forward call references

**Pass 2 — Reverse Call Graph:**

- Build indexes: `pkgIndex` maps `"dir.FuncName"` → chunk references
- Build `suffixToDir` for O(1) package suffix matching
- For each chunk's `Calls`, find the target chunk and add the caller to its `CalledBy`

**Pass 3 — Describe + Embed + Store:**

- For each file, send content + relationship context to the local LLM for description generation
- Build embedding text (`signature + "\n" + description`) for each chunk
- Batch embed via the embedding container
- Upsert chunks with embeddings into LanceDB
- Update file hash cache

### Change Detection

- File hashes stored in `rag_file_hashes.json` in the project data directory
- On re-index, files with unchanged hashes are skipped entirely
- Schema version changes trigger a full re-index (drop and recreate table)
- Force flag available for manual full re-index

### Indexing Triggers for sirtopham

- **Full index:** on `yard init` or first run against a project
- **Incremental index:** via `yard index` without `--force`; file-hash change detection decides what gets re-parsed and re-embedded
- **Manual re-index:** via `yard index`
- **No startup or conversation auto-reindex today:** `serve` does not trigger indexing, so operators should run `yard index` explicitly before expecting fresh retrieval

## Component: Searcher

### Basic Search

Embed the query → vector search → return top-K results with cosine similarity scores.

### Work-Order-Aware Search (Enriched)

The sophisticated search path used by topham's pipeline phases:

1. **Query expansion:** Extract multiple search queries from different fields (title, target module, acceptance criteria, known files)
2. **Multi-query execution:** Run each expanded query with `topK=10`
3. **Deduplication + re-ranking:** Chunks that match multiple queries get higher rank (hit count). Ties broken by best similarity score.
4. **Budget allocation:** 60% of max results for direct vector hits, 40% for dependency hops
5. **One-hop call graph expansion:** For each direct hit, look up functions it calls and functions that call it via the store's name index

### Adaptation for sirtopham

topham's `SearchForWorkOrder` is pipeline-specific (work orders, acceptance criteria). sirtopham needs a conversational equivalent — the turn analyzer ([[06-context-assembly]]) determines what to search for, and the searcher executes it. The multi-query expansion and dependency hop patterns should be preserved; only the input structure changes.

## Component: Structural Graph

Separate from RAG. Stored in SQLite with its own analyzers.

### Analyzers

- **Go:** Uses `go/packages` with full type info for call graph, type references, interface implementations
- **Python:** AST-based analysis for imports, calls, class hierarchies
- **TypeScript:** External Node.js analyzer script (`ts-analyzer/analyze.ts`) for imports, calls, type references

### Blast Radius Query

Given a target symbol, returns:

- **Upstream:** functions/types that call or reference the target (callers)
- **Downstream:** functions/types the target calls or references (callees)
- **Interfaces:** interfaces the target type implements
- Configurable depth, budget, and confidence thresholds

### Use in sirtopham

When the user asks "what would break if I change function X", the structural graph provides the answer. When context assembly identifies specific files being modified, the graph can automatically pull in structurally related code that might need updating.

- **Incremental indexing via git-aware/manual change detection** (incremental runs happen when `yard index` is invoked without `--force`)
- **Explicit operator-controlled indexing before `serve`**
- **Turn-aware search** (replacing work-order-aware search with conversational query extraction)
- **Combined RAG + graph queries** driven by the context assembly turn analyzer

## Open Questions

- **Description model selection:** Which local model produces the best descriptions for the embedding pipeline? Worth A/B testing. A better description model could be the single highest-impact quality improvement.
- **Embedding model alternatives:** nomic-embed-code is good, but code embedding models are evolving fast. Worth periodic re-evaluation.
- **Cross-language queries:** How well does semantic search work when the query is in English but the code is in Go/Python/TS? The description layer should bridge this, but worth testing.
- **Index size scaling:** How large does the LanceDB index get for a 100k+ line codebase? Is query latency still acceptable?
